# Red Tetris — Plan Technique Detaille du Backend

> Node.js | TypeScript | Socket.IO | Express.js
> Programmation Orientee Objet | Classes obligatoires | Zero persistence

---

## 1. Principes Fondamentaux du Backend

### 1.1 Regles Imperatives
- **Programmation Orientee Objet** : le code serveur doit utiliser des classes/prototypes
- **Classes obligatoires minimum** : `Player`, `Piece`, `Game`
- **Communication** : Socket.IO pour le temps reel bidirectionnel
- **Serveur HTTP** : doit servir les fichiers statiques (index.html, bundle.js) via HTTP
- **Zero persistence** : tout l'etat est en memoire — pas de base de donnees
- **TypeScript** pour le typage fort et la robustesse

### 1.2 Responsabilites du Serveur
Le serveur est le **garant de la verite** pour :
- La gestion des rooms (creation, suppression, joueurs)
- La distribution des pieces (sequence unique partagee par tous les joueurs d'une room)
- La gestion du cycle de vie des parties (attente, en cours, terminee)
- La transmission des penalites entre joueurs
- La diffusion des spectres entre joueurs
- Le transfert du role de host
- Le verrouillage des rooms une fois la partie demarree

Le serveur ne gere **pas** :
- La logique de mouvement des pieces (c'est le client qui gere)
- Le rendu ou l'affichage
- La persistence des donnees

## 3. Point d'Entree — index.ts

### 3.1 Responsabilites
Le fichier `index.ts` est le point d'entree unique du serveur. Il orchestre l'initialisation de tous les modules.

**Etapes d'initialisation** :
1. Chargement des variables d'environnement (dotenv)
2. Creation de l'application Express
3. Configuration du serveur HTTP (`http.createServer(app)`)
4. Attachement de Socket.IO au serveur HTTP avec configuration CORS
5. Configuration du serving de fichiers statiques (index.html, bundle.js du client)
6. Instanciation du `GameManager` (singleton)
7. Configuration de l'evenement `connection` Socket.IO → enregistrement des handlers
8. Demarrage du serveur sur le port configure

### 3.2 Configuration Socket.IO
- **CORS** : autoriser l'origine du client (configurable via env)
- **Transports** : websocket en priorite, polling en fallback
- **Ping timeout / interval** : configuration pour detecter les deconnexions

### 3.3 Serving de Fichiers Statiques
- Express sert le dossier `client/build/` (ou `client/out/` selon la config Next.js)
- Route catch-all pour rediriger toutes les routes vers `index.html` (SPA)
- Le bundle.js est servi depuis le meme serveur HTTP

## 4. Classe Player

### 4.1 Proprietes
| Propriete | Type | Description |
|-----------|------|-------------|
| `id` | `string` | Identifiant unique du joueur (= `socket.id`) |
| `name` | `string` | Nom d'affichage choisi par le joueur |
| `roomName` | `string` | Nom de la room dans laquelle il est |
| `socket` | `Socket` | Reference directe au socket pour emissions ciblees |
| `isHost` | `boolean` | Est-il l'hote de la room ? (premier joueur a rejoindre) |
| `isAlive` | `boolean` | Est-il encore en jeu ? (pas elimine) |
| `pieceIndex` | `number` | Index de la prochaine piece a recuperer dans la sequence partagee |

### 4.2 Methodes
**`constructor(socket: Socket, name: string, roomName: string)`**
- Initialise toutes les proprietes
- `id` = `socket.id`
- `isHost` = false (sera defini par la Game)
- `isAlive` = true
- `pieceIndex` = 0

**`emit(event: string, data: unknown): void`**
- Envoie un evenement uniquement a ce joueur via son socket
- Wrapper type pour centraliser les emissions individuelles

**`resetForNewGame(): void`**
- Remet `isAlive` a true
- Remet `pieceIndex` a 0
- Utilise pour le restart d'une partie sans deconnexion

### 4.3 Points d'Attention
- La reference au `socket` permet d'envoyer des messages cibles sans passer par la room Socket.IO
- Le `pieceIndex` est incremente a chaque appel `REQUEST_PIECE` — chaque joueur avance a son rythme dans la meme sequence
- Le player est identifie par son `socket.id`, qui est unique par connexion

## 5. Classe Piece

### 5.1 Proprietes
| Propriete | Type | Description |
|-----------|------|-------------|
| `type` | `PieceType` | Type de tetrimino : 'I' | 'O' | 'T' | 'S' | 'Z' | 'J' | 'L' |
| `rotation` | `number` | Etat de rotation initial (0) |
| `position` | `{ x: number, y: number }` | Position de spawn sur le plateau |

### 5.2 Methodes
**`constructor(type?: PieceType)`**
- Si type fourni : utilise ce type
- Si type non fourni : genere un type aleatoire via `Piece.randomType()`
- Position de spawn : centree horizontalement (x = 3 ou 4 selon la piece), y = 0

**`static randomType(): PieceType`**
- Retourne un type aleatoire parmi les 7 tetriminos
- Distribution uniforme

**`static spawnPosition(type: PieceType): Position`**
- Retourne la position de spawn standard pour un type donne
- Centree horizontalement sur le plateau de 10 colonnes

**`toDTO(): IPiece`**
- Serialise la piece en objet simple pour l'envoi via Socket.IO
- Retourne `{ type, rotation, position }`

### 5.3 Points d'Attention
- Les pieces sont generees **uniquement cote serveur** et distribuees aux clients
- La piece envoyee au client contient seulement le type et la position de spawn — le client calcule les rotations et mouvements localement
- Toutes les pieces d'une meme sequence sont generees au demarrage de la partie

## 6. Classe Game

### 6.1 Proprietes
| Propriete | Type | Description |
|-----------|------|-------------|
| `roomName` | `string` | Identifiant unique de la room/partie |
| `players` | `Map<string, Player>` | Map des joueurs (cle = socket.id) |
| `status` | `'waiting' | 'playing' | 'ended'` | Statut actuel de la partie |
| `pieces` | `Piece[]` | Sequence de pieces pre-generee et partagee |
| `host` | `Player | null` | Reference au joueur hote actuel |

### 6.2 Methodes de Gestion des Joueurs

**`addPlayer(player: Player): void`**
- Ajoute le joueur a la `Map` players
- Si c'est le premier joueur : le designe comme host (`player.isHost = true`, `this.host = player`)
- Si la partie est en cours (`status === 'playing'`) : refuse l'ajout (emit erreur au joueur)
- Notifie tous les joueurs de la room via `PLAYER_JOINED`

**`removePlayer(socketId: string): void`**
- Retire le joueur de la Map
- Si le joueur retire etait le host : appelle `transferHost()`
- Si le joueur retire etait en jeu (`isAlive`) : appelle `checkWinCondition()`
- Si la room est vide apres le retrait : signale au GameManager de supprimer la room
- Notifie les joueurs restants via `PLAYER_LEFT`

**`transferHost(): void`**
- Selectionne le premier joueur restant dans la Map comme nouveau host
- Met a jour les proprietes `isHost` de l'ancien et du nouveau host
- Emet `HOST_CHANGED` a tous les joueurs de la room

### 6.3 Methodes de Gestion de la Partie

**`start(): void`**
- Verifie que le status est 'waiting'
- Genere la sequence de pieces partagee via `pieceGenerator` (ex: 1000 pieces)
- Passe le status a 'playing'
- Reinitialise tous les joueurs (`resetForNewGame`)
- Emet `GAME_STARTED` a tous les joueurs

**`getNextPiece(player: Player): Piece`**
- Recupere la piece a l'index `player.pieceIndex` dans la sequence `this.pieces`
- Incremente `player.pieceIndex`
- Si l'index depasse la taille de la sequence : genere des pieces supplementaires (extension dynamique)
- Retourne la piece correspondante

**`applyPenalty(sourcePlayer: Player, linesCleared: number): void`**
- Calcule le nombre de penalites : `linesCleared - 1`
- Si penalites > 0 : emet `PENALTY_LINES` a **tous les joueurs sauf le source**
- Ne penalise que les joueurs encore vivants (`isAlive === true`)

**`updateSpectrum(sourcePlayer: Player, spectrum: number[]): void`**
- Recoit le spectre du joueur source
- Emet `SPECTRUM_UPDATE` a **tous les joueurs de la room sauf le source**
- Payload : `{ playerName: sourcePlayer.name, spectrum }`

**`eliminatePlayer(player: Player): void`**
- Met `player.isAlive` a false
- Emet `PLAYER_ELIMINATED` a tous les joueurs de la room
- Appelle `checkWinCondition()`

**`checkWinCondition(): void`**
- Compte le nombre de joueurs encore vivants (`isAlive === true`)
- Si 1 seul joueur vivant (et plus d'1 joueur au total) : c'est le vainqueur
  - Passe le status a 'ended'
  - Emet `GAME_OVER` avec `{ winner: lastPlayer.name }` a tous les joueurs
- Si 0 joueur vivant (mode solo ou elimination simultanee) :
  - Passe le status a 'ended'
  - Emet `GAME_OVER` avec `{ winner: null }` (pas de vainqueur)

**`restart(): void`**
- Verifie que le status est 'ended'
- Genere une nouvelle sequence de pieces
- Reinitialise tous les joueurs
- Passe le status a 'waiting' (retour au lobby)
- Ou directement a 'playing' si restart immediat (selon la preference)

### 6.4 Methodes Utilitaires

**`getPlayerBySocketId(socketId: string): Player | undefined`**
- Raccourci pour `this.players.get(socketId)`

**`getAlivePlayers(): Player[]`**
- Retourne la liste des joueurs avec `isAlive === true`

**`getPlayerCount(): number`**
- Retourne le nombre total de joueurs dans la room

**`getRoomState(): IRoomState`**
- Serialise l'etat de la room pour l'envoi au client
- Retourne : `{ players: [...], status, host: hostName }`

**`broadcastToAll(event: string, data: unknown): void`**
- Emet un evenement a **tous** les joueurs de la room

**`broadcastToOthers(excludeSocketId: string, event: string, data: unknown): void`**
- Emet un evenement a tous les joueurs **sauf** celui specifie

## 7. GameManager — Gestionnaire Global

### 7.1 Proprietes
| Propriete | Type | Description |
|-----------|------|-------------|
| `games` | `Map<string, Game>` | Map de toutes les parties actives (cle = roomName) |

### 7.2 Methodes

**`getOrCreateGame(roomName: string): Game`**
- Si une Game existe pour ce roomName : la retourne
- Sinon : cree une nouvelle Game, la stocke dans la Map, et la retourne
- C'est le point d'entree pour tout joueur qui rejoint une room

**`getGame(roomName: string): Game | undefined`**
- Retourne la Game associee a une room, ou undefined si elle n'existe pas

**`removeGame(roomName: string): void`**
- Supprime la Game de la Map
- Appele quand la room est vide (tous les joueurs ont quitte)
- Libere la memoire

**`getActiveGamesCount(): number`**
- Retourne le nombre de parties actives (utile pour le monitoring)

**`getAllGames(): Game[]`**
- Retourne toutes les parties (utile pour le debug)

### 7.3 Pattern Singleton
- Le GameManager est instancie **une seule fois** dans `index.ts`
- Il est passe par reference aux handlers socket
- Toutes les operations sur les rooms transitent par lui
- Il garantit qu'une seule instance de Game existe par roomName

## 8. Gestion Socket.IO (`socket/`)

### 8.1 registerEvents.ts — Enregistrement des Listeners
**Principe** : Pour chaque nouvelle connexion socket, on enregistre tous les handlers d'evenements.

**Evenement `connection`** :
- Quand un client se connecte, Socket.IO fournit un objet `socket`
- On enregistre les listeners pour tous les evenements client → serveur
- On enregistre le handler de `disconnect`

**Structure** :
- Recoit le `socket` et le `gameManager` en parametres
- Enregistre chaque evenement avec son handler correspondant

### 8.2 handlers.ts — Traitement des Evenements Entrants

**`handleJoinRoom(socket, gameManager, data: { room, playerName })`**
1. Valide les inputs (room non-vide, playerName non-vide, caracteres autorises)
2. Appelle `gameManager.getOrCreateGame(room)` pour obtenir la Game
3. Verifie que la partie n'est pas en cours (sinon : emission d'erreur)
4. Verifie que le nom n'est pas deja pris dans la room (sinon : emission d'erreur)
5. Cree un nouvel objet `Player(socket, playerName, room)`
6. Appelle `game.addPlayer(player)`
7. Fait rejoindre le socket a la room Socket.IO (`socket.join(room)`)
8. Emet `ROOM_STATE` au joueur qui vient de rejoindre
9. Emet `PLAYER_JOINED` a tous les autres joueurs de la room

**`handleStartGame(socket, gameManager, data: { room })`**
1. Recupere la Game via `gameManager.getGame(room)`
2. Verifie que le socket correspond au host
3. Appelle `game.start()`
4. Le start emet `GAME_STARTED` a tous les joueurs

**`handleRequestPiece(socket, gameManager, data: { room })`**
1. Recupere la Game et le Player
2. Appelle `game.getNextPiece(player)`
3. Emet `NEW_PIECE` au joueur demandeur avec la piece serialisee

**`handleUpdateSpectrum(socket, gameManager, data: { room, spectrum })`**
1. Recupere la Game et le Player
2. Valide le spectre (tableau de 10 nombres)
3. Appelle `game.updateSpectrum(player, spectrum)`

**`handleLinesCleared(socket, gameManager, data: { room, count })`**
1. Recupere la Game et le Player
2. Valide le count (nombre entier positif)
3. Appelle `game.applyPenalty(player, count)`

**`handleGameOverPlayer(socket, gameManager, data: { room })`**
1. Recupere la Game et le Player
2. Appelle `game.eliminatePlayer(player)`

**`handleRestartGame(socket, gameManager, data: { room })`**
1. Recupere la Game
2. Verifie que le socket correspond au host
3. Appelle `game.restart()`

**`handleDisconnect(socket, gameManager)`**
1. Determine la room du joueur (stockee dans les donnees associees au socket ou dans le Player)
2. Recupere la Game
3. Appelle `game.removePlayer(socket.id)`
4. Si la room est vide : appelle `gameManager.removeGame(roomName)`

### 8.3 emitters.ts — Fonctions d'Emission
Fonctions utilitaires pour centraliser et typer les emissions serveur → client.

**`emitToPlayer(player: Player, event: string, data: unknown): void`**
- Emet un evenement directement au socket d'un joueur specifique

**`emitToRoom(io: Server, roomName: string, event: string, data: unknown): void`**
- Emet un evenement a tous les sockets d'une room Socket.IO

**`emitToRoomExcept(socket: Socket, roomName: string, event: string, data: unknown): void`**
- Emet un evenement a tous les sockets d'une room sauf l'emetteur (`socket.to(room).emit(...)`)

**`emitError(player: Player, message: string): void`**
- Emet un evenement d'erreur specifique au joueur (ex: "Room en cours", "Nom deja pris")

## 9. Generateur de Pieces (`utils/pieceGenerator.ts`)

### 9.1 Principe
Le generateur de pieces est le composant qui garantit l'equite du jeu multijoueur. Tous les joueurs d'une meme room doivent recevoir les memes pieces dans le meme ordre.

### 9.2 Fonctionnement

**`generatePieceSequence(count: number): Piece[]`**
- Genere un tableau de `count` pieces aleatoires
- Chaque piece est instanciee via `new Piece()` avec un type aleatoire
- La sequence est stockee dans la Game et indexee par joueur via `pieceIndex`

**Approche alternative — Bag System (7-bag randomizer)** :
- Plus fidele aux regles officielles du Tetris moderne
- Melange les 7 types de pieces dans un "sac"
- Chaque sac contient exactement 1 de chaque type
- Quand un sac est vide, un nouveau sac est melange
- Garantit une distribution equitable des pieces

### 9.3 Extension Dynamique
- Si un joueur atteint la fin de la sequence pre-generee, le serveur genere des pieces supplementaires
- La generation supplementaire est ajoutee a la meme sequence pour maintenir la coherence

### 9.4 Serialisation
- Les pieces sont transmises aux clients via `NEW_PIECE` en format DTO (plain object)
- Format : `{ type: string, rotation: number, position: { x: number, y: number } }`

## 10. Validation des Inputs (`utils/validation.ts`)

### 10.1 Validation du Nom de Room
- Non-vide
- Longueur entre 1 et 20 caracteres
- Caracteres autorises : alphanumeriques, tirets, underscores
- Pas de caracteres speciaux ou espaces

### 10.2 Validation du Nom de Joueur
- Non-vide
- Longueur entre 1 et 15 caracteres
- Caracteres autorises : alphanumeriques, tirets, underscores
- Unicite dans la room (pas de doublon)

### 10.3 Validation du Spectre
- Doit etre un tableau de exactement 10 nombres
- Chaque valeur entre 0 et 20 (hauteur max du plateau)

### 10.4 Validation du Count (lignes effacees)
- Nombre entier positif
- Entre 1 et 4 (maximum de lignes effacables en un coup)

## 13. Flux d'Execution Backend — Cycle de Vie

### 13.1 Demarrage du Serveur
1. Chargement de la configuration (.env)
2. Initialisation Express + HTTP + Socket.IO
3. Instanciation du GameManager
4. Le serveur ecoute sur le port configure
5. En attente de connexions

### 13.2 Connexion d'un Joueur
1. Un client se connecte → evenement `connection`
2. Enregistrement de tous les listeners sur le socket
3. Le client emet `JOIN_ROOM` avec `{ room, playerName }`
4. Le handler valide les inputs
5. `GameManager.getOrCreateGame(room)` cree ou recupere la Game
6. Un objet Player est cree
7. `game.addPlayer(player)` ajoute le joueur
8. `ROOM_STATE` est emis au nouveau joueur
9. `PLAYER_JOINED` est emis aux autres joueurs de la room

### 13.3 Demarrage d'une Partie
1. Le host emet `START_GAME`
2. Le handler verifie que c'est bien le host
3. `game.start()` :
   - Genere la sequence de pieces (1000+)
   - Reinitialise les joueurs
   - Passe en status 'playing'
4. `GAME_STARTED` est emis a tous les joueurs
5. Les clients emettent `REQUEST_PIECE` pour recevoir leur premiere piece

### 13.4 Boucle de Jeu (vue serveur)
Le serveur ne gere pas de boucle de jeu a proprement parler. Il reagit aux evenements des clients :

**Chaque cycle de pose de piece** :
1. Le client emet `REQUEST_PIECE` → le serveur repond avec `NEW_PIECE`
2. Le client emet `UPDATE_SPECTRUM` → le serveur broadcast aux adversaires
3. Si le client a efface des lignes : `LINES_CLEARED` → le serveur envoie `PENALTY_LINES` aux adversaires
4. Si le client est elimine : `GAME_OVER_PLAYER` → le serveur broadcast `PLAYER_ELIMINATED` et verifie la condition de victoire

### 13.5 Deconnexion d'un Joueur
1. Evenement `disconnect` detecte
2. Le handler identifie la room du joueur
3. `game.removePlayer(socketId)` retire le joueur
4. Si le joueur etait host : `transferHost()` designe un nouveau host
5. Si le joueur etait vivant en partie : `checkWinCondition()`
6. `PLAYER_LEFT` est emis aux joueurs restants
7. Si la room est vide : `gameManager.removeGame(roomName)`

### 13.6 Fin de Partie
1. `checkWinCondition()` detecte un seul joueur survivant
2. Le status passe a 'ended'
3. `GAME_OVER` avec `{ winner }` est emis a tous
4. Le host peut emettre `RESTART_GAME` pour relancer

## 14. Gestion de la Memoire et des Erreurs

### 14.1 Nettoyage Memoire
- Quand une room est vide, elle est supprimee du GameManager
- La sequence de pieces est garbage collected avec la Game
- Les references aux sockets sont nettoyees au retrait du Player
- Pas de fuite memoire sur les listeners (cleanup a la deconnexion)

### 14.2 Gestion des Cas Limites
- **Joueur qui rejoint pendant une partie** : refus avec emission `ERROR`
- **Nom deja pris dans la room** : refus avec emission `ERROR`
- **Host qui quitte en pleine partie** : transfert automatique du host
- **Tous les joueurs deconnectes** : suppression de la room
- **Deconnexion brutale** (perte de connexion) : le timeout Socket.IO declenche `disconnect`
- **Requete piece hors limites** : extension dynamique de la sequence
- **Double-emission** (client envoie 2x le meme event) : idempotence quand possible

### 14.3 Securite des Inputs
- Tous les inputs sont valides avant traitement
- Les event handlers sont wrapes dans des try/catch pour eviter le crash du serveur
- Les erreurs sont logguees cote serveur et un message generique est renvoye au client
- Pas de donnees sensibles exposees dans les messages d'erreur

## 15. Strategie de Tests Backend

### 15.1 Tests de la Classe Player — Priorite Haute
- **Construction** : verifier que toutes les proprietes sont initialisees correctement
- **emit()** : verifier que le socket recoit bien l'evenement (mock socket)
- **resetForNewGame()** : verifier la reinitialisation de `isAlive` et `pieceIndex`

### 15.2 Tests de la Classe Piece — Priorite Haute
- **Construction sans type** : verifie qu'un type aleatoire est genere parmi les 7 valides
- **Construction avec type** : verifie que le type fourni est utilise
- **spawnPosition()** : verifier le centrage horizontal pour chaque type
- **toDTO()** : verifier la serialisation correcte

### 15.3 Tests de la Classe Game — Priorite Haute
**Gestion des joueurs** :
- `addPlayer()` : ajout correct, designation du premier comme host
- `addPlayer()` pendant une partie en cours : refus
- `removePlayer()` : retrait correct, nettoyage
- `removePlayer()` du host : transfert automatique
- `transferHost()` : nouveau host correctement designe

**Gestion de la partie** :
