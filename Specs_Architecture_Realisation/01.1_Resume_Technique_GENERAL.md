# Red Tetris — Resume Technique General

> Projet 42 | Version 5.2 | Full Stack JavaScript/TypeScript
> Tetris Multijoueur en Reseau — "Tetris Network with Red Pelicans Sauce"

---

## 1. Presentation du Projet

Red Tetris est un jeu Tetris multijoueur en reseau, developpe integralement en JavaScript/TypeScript selon une architecture Full Stack. Le projet repose sur une communication temps reel bidirectionnelle entre un client SPA (Single Page Application) et un serveur Node.js, via le protocole Socket.IO.

L'objectif pedagogique est triple : maitriser la programmation fonctionnelle cote client, la programmation orientee objet cote serveur, et la conception d'un protocole reseau asynchrone temps reel.

## 2. Stack Technique Imposee

### 2.1 Frontend (Client)
- **Langage** : TypeScript (superset strict de JavaScript)
- **Framework** : Next.js (React)
- **Architecture** : SPA — Single Page Application
- **Communication** : Socket.IO (socket.io-client)
- **Styling** : Tailwind CSS (approche utilitaire, design minimaliste moderne)
- **Gestion d'etat** : Redux Toolkit (ou Zustand — a definir)
- **Layout CSS** : Grid et Flexbox exclusivement
- **Paradigme** : Programmation fonctionnelle (pas de `this`, fonctions pures pour la logique de jeu)

### 2.2 Backend (Serveur)
- **Langage** : TypeScript
- **Runtime** : Node.js
- **Communication** : Socket.IO
- **Serveur HTTP** : Express.js (ou http natif)
- **Paradigme** : Programmation Orientee Objet (classes/prototypes)
- **Persistence** : Aucune (tout en memoire)

### 2.3 Partage Client/Serveur
- **Dossier shared/** : Types TypeScript, constantes du jeu, noms des evenements Socket.IO
- Source de verite unique pour le protocole reseau

## 3. Contraintes Techniques Obligatoires

### 3.1 Interdictions Strictes (Client)
- **`this`** : Interdit dans tout le code client (exception : sous-classes de `Error`)
- **jQuery** : Interdit (et toute lib de manipulation DOM)
- **Canvas** : Interdit
- **SVG** : Interdit
- **`<TABLE />`** : Interdit en HTML
- **Manipulation DOM directe** : Non necessaire avec React

### 3.2 Obligations Client
- Utilisation exclusive de **grid** et **flexbox** pour les layouts
- Logique de jeu (plateau, pieces, mouvements) implementee en **fonctions pures**
- Application **SPA** : un seul `index.html`, un `bundle.js`, pas d'echange de pages HTML
- Composants **fonctionnels** React (hooks, pas de classes)

### 3.3 Obligations Serveur
- Approche **orientee objet** avec classes/prototypes
- Classes obligatoires minimum : `Player`, `Piece`, `Game`
- Serveur Node.js avec Socket.IO
- Doit servir les fichiers statiques (index.html, bundle.js) via HTTP

### 3.4 Obligations Generales
- **Tests unitaires** : couverture minimum 70% statements/functions/lines, 50% branches
- **Securite** : aucun credential, API key ou variable d'env dans le repo (.env gitignore)
- **Memes pieces** : tous les joueurs d'une room recoivent la meme sequence de pieces dans le meme ordre, aux memes positions et coordonnees

## 4. Regles de Jeu — Specifications Fonctionnelles

### 4.1 Le Plateau
- Grille de **10 colonnes x 20 lignes**
- Representation interne : tableau 2D (0 = vide, 1-7 = type de piece, 8 = ligne de penalite)
- Les lignes de penalite sont **indestructibles** (ne peuvent pas etre effacees)

### 4.2 Les 7 Tetriminos
| Piece | Forme | Couleur |
|-------|-------|---------|
| I | Barre horizontale de 4 | Cyan |
| O | Carre 2x2 | Jaune |
| T | Forme en T | Violet |
| S | Forme en S | Vert |
| Z | Forme en Z | Rouge |
| J | Forme en L inverse | Bleu |
| L | Forme en L | Orange |

Chaque piece possede **4 etats de rotation** conformes aux regles originales du Tetris.

### 4.3 Controles Joueur
| Touche | Action |
|--------|--------|
| Fleche Gauche | Deplacement horizontal gauche |
| Fleche Droite | Deplacement horizontal droit |
| Fleche Haut | Rotation de la piece |
| Fleche Bas | Soft drop (descente acceleree) |
| Barre Espace | Hard drop (chute instantanee) |

### 4.4 Mecanique de Chute
- Les pieces descendent automatiquement a **vitesse constante** (intervalle configurable)
- Quand une piece touche le bas ou une autre piece, elle reste mobile **un frame supplementaire** (last-moment adjustment) avant d'etre figee
- Apres verrouillage : verification des lignes completes, puis generation de la piece suivante

### 4.5 Effacement de Lignes et Penalites Multijoueur
- Une ligne **complete** (toutes les cellules non-nulles) disparait
- Les lignes superieures descendent pour combler
- Systeme de penalite multijoueur :
  - 1 ligne effacee → 0 penalite
  - 2 lignes effacees → 1 ligne de penalite aux adversaires
  - 3 lignes effacees → 2 lignes de penalite
  - 4 lignes effacees (Tetris) → 3 lignes de penalite
- Les lignes de penalite s'ajoutent **en bas** du plateau adverse et sont **indestructibles**

### 4.6 Conditions de Fin
- **Elimination** : un joueur est elimine quand une nouvelle piece ne peut pas apparaitre a sa position de spawn (plateau sature)
- **Victoire** : le dernier joueur survivant gagne
- **Solo** : la partie se termine a l'elimination du joueur unique
- **Pas de systeme de score** dans le mandatory (le dernier survivant gagne)

### 4.7 Spectre (Spectrum)
- Chaque joueur voit une representation compacte des plateaux adverses
- Le spectre = tableau de 10 valeurs (une par colonne) representant la hauteur du bloc le plus haut de chaque colonne
- Mise a jour **temps reel** via Socket.IO
- Affiche le nom et le statut (vivant/elimine) de chaque adversaire

## 5. Gestion des Parties — Game Management

### 5.1 Acces au Jeu via URL
- Format : `http://<server>:<port>/<room>/<player_name>`
- `room` : identifiant de la salle de jeu
- `player_name` : nom du joueur
- Utilisation du routing dynamique Next.js : `app/[room]/[player]/page.tsx`

### 5.2 Systeme de Rooms
- **Host** : le premier joueur a rejoindre une room devient l'hote
- Le host est le **seul** a pouvoir demarrer et relancer la partie
- Si le host quitte, le role est **transfere** a un autre joueur present
- **Aucun nouveau joueur** ne peut rejoindre une partie en cours (jusqu'au prochain round)
- **Plusieurs parties concurrentes** sont supportees (plusieurs rooms simultanees)
- Un joueur seul peut jouer (mode solo)

### 5.3 Flux d'une Partie Type
1. Les joueurs rejoignent la room via l'URL → ecran Lobby
2. Le host voit un bouton "Demarrer" — les autres joueurs attendent
3. Le host lance la partie → le serveur genere une sequence de pieces partagee
4. Chaque joueur joue sur son propre plateau avec les memes pieces
5. Les spectres des adversaires se mettent a jour en temps reel
6. Les lignes effacees generent des penalites pour les adversaires
7. Les joueurs sont elimines un par un
8. Le dernier survivant gagne → ecran de victoire
9. Le host peut relancer une nouvelle partie

## 6. Protocole Reseau — Socket.IO

### 6.1 Evenements Client → Serveur
| Evenement | Donnees | Description |
|-----------|---------|-------------|
| `JOIN_ROOM` | `{ room, playerName }` | Rejoindre une room |
| `START_GAME` | `{ room }` | Le host demarre la partie |
| `REQUEST_PIECE` | `{ room }` | Demander la prochaine piece |
| `UPDATE_SPECTRUM` | `{ room, spectrum: number[] }` | Envoyer son spectre actualise |
| `LINES_CLEARED` | `{ room, count }` | Signaler des lignes effacees |
| `GAME_OVER_PLAYER` | `{ room }` | Signaler son elimination |
| `RESTART_GAME` | `{ room }` | Le host relance une partie |

### 6.2 Evenements Serveur → Client
| Evenement | Donnees | Description |
|-----------|---------|-------------|
| `ROOM_STATE` | `{ players, status, isHost }` | Etat complet de la room |
| `PLAYER_JOINED` | `{ playerName }` | Un joueur a rejoint |
| `PLAYER_LEFT` | `{ playerName }` | Un joueur a quitte |
| `HOST_CHANGED` | `{ newHost }` | Nouveau host designe |
| `GAME_STARTED` | `{}` | La partie commence |
| `NEW_PIECE` | `{ piece: IPiece }` | Piece suivante envoyee |
| `SPECTRUM_UPDATE` | `{ playerName, spectrum }` | Spectre d'un adversaire mis a jour |
| `PENALTY_LINES` | `{ count }` | Lignes de penalite recues |
| `PLAYER_ELIMINATED` | `{ playerName }` | Un joueur a ete elimine |
| `GAME_OVER` | `{ winner }` | Partie terminee, annonce du vainqueur |
| `ERROR` | `{ message: string }` | Erreur serveur (room en cours, nom deja pris, input invalide) |

### 6.3 Principes Cles du Protocole
- La **logique de mouvement** est traitee **integralement cote client** (fonctions pures)
- Le serveur n'est **pas notifie** a chaque deplacement de piece
- Le serveur est notifie uniquement lors de : pose de piece, lignes effacees, mise a jour du spectre, elimination
- La **sequence de pieces** est generee cote serveur et distribuee a la demande via `REQUEST_PIECE` / `NEW_PIECE`
- Chaque joueur avance dans la sequence a son propre rythme via son `pieceIndex`

## 7. Architecture Applicative — Vue d'Ensemble

### 7.1 Approche Component-First (Frontend)
L'architecture frontend suit une approche **strictement componentisee** :
- Les **Pages** ne gerent que l'UI et l'orchestration des composants (aucune logique metier)
- Toute la **logique metier** est encapsulee dans les composants
- Chaque composant possede son propre **dossier** contenant :
  - 1 fichier Component (logique + rendu)
  - 1 fichier Layout (mise en page propre au composant)
- Les Layouts de pages sont eux-memes des compositions de composants

**Exemple d'organisation** :
```
Components/
  CreateRoomComponents/
    CreateRoom_Component.tsx    ← Logique metier + rendu
    Layout_CreateRoom.tsx       ← Layout specifique du composant
  GameBoardComponents/
    GameBoard_Component.tsx
    Layout_GameBoard.tsx
  SpectrumComponents/
    Spectrum_Component.tsx
    Layout_Spectrum.tsx
```

### 7.2 Separation des Responsabilites
| Couche | Responsabilite |
|--------|---------------|
| **Pages** (Next.js App Router) | Orchestration des composants, routing, extraction des params URL |
| **Layouts** | Mise en page, positionnement des composants |
| **Components** | Logique metier, gestion d'etat local, emission/ecoute socket |
| **Hooks** | Logique reutilisable (socket, keyboard, game loop) |
| **Game (fonctions pures)** | Logique de jeu sans effets de bord (plateau, pieces, mouvements) |
| **Store (Redux)** | Etat global de l'application |
| **Socket** | Communication temps reel avec le serveur |

### 7.3 Architecture Serveur (OOP)
| Classe/Module | Responsabilite |
|--------------|---------------|
| **Player** | Representation d'un joueur connecte (socket, nom, room, statut) |
| **Piece** | Representation d'un tetrimino (type, position de spawn) |
| **Game** | Gestion d'une partie (joueurs, sequence de pieces, statut, penalites) |
| **GameManager** | Gestionnaire singleton de toutes les parties concurrentes |
| **Socket Handlers** | Traitement des evenements entrants |
| **Socket Emitters** | Emission des evenements sortants |
| **Piece Generator** | Generation de sequences de pieces partagees |

## 9. Design et UI — Approche Visuelle

### 9.1 Philosophie Design
- **Minimalisme colore** inspire d'Instagram
- **Glass Effect** (glassmorphism) sur les panneaux et overlays
- Palette de couleurs sobre avec accents vifs pour les tetriminos
- Animations fluides et transitions elegantes
- Focus sur la **Consumer Experience** (UX intuitive, responsive)

### 9.2 Framework CSS
- **Tailwind CSS** comme framework utilitaire principal
- Classes utilitaires pour tous les styles
- Utilisation de `backdrop-blur`, `bg-opacity`, `rounded`, `shadow` pour le glass effect
- Grid CSS pour le plateau de jeu
- Flexbox pour les layouts de page

### 9.3 Ecrans Principaux
1. **Page d'accueil / Lobby** : creation ou rejoindre une room, liste des joueurs
2. **Ecran de jeu** : plateau principal, piece suivante, spectres adversaires, infos
3. **Overlay Game Over** : ecran de fin (defaite ou victoire), option de relancer
4. **Transitions** : animations entre les ecrans (SPA, pas de rechargement)

## 10. Tests — Strategie de Couverture

### 10.1 Seuils Obligatoires
| Metrique | Minimum |
|----------|---------|
| Statements | 70% |
| Functions | 70% |
| Lines | 70% |
| Branches | 50% |

### 10.2 Frameworks de Test
- **Jest** + **ts-jest** pour TypeScript
- **@testing-library/react** pour les composants React
- **socket.io-mock** ou socket.io-client en mode test pour les tests d'integration

### 10.3 Priorites de Test
1. **Priorite Haute** : Fonctions pures de logique de jeu (board, pieces, movement, lines) — facilement testables, haut ratio couverture/effort
2. **Priorite Haute** : Classes serveur (Player, Piece, Game, GameManager)
3. **Priorite Moyenne** : Composants React (rendu correct, props, conditions)
4. **Priorite Basse** : Integration Socket.IO (flux complet client-serveur)

## 12. Points de Vigilance

1. **Ne jamais utiliser `this` dans le code client** (sauf sous-classes Error)
2. **Ne jamais stocker de secrets dans le repo Git** (.env + .gitignore)
3. **Tous les joueurs d'une room recoivent les memes pieces dans le meme ordre**
4. **Aucun nouveau joueur ne peut rejoindre une partie en cours**
5. **Le host est le seul a pouvoir demarrer/relancer la partie**
6. **La logique de mouvement est 100% cote client** — le serveur ne gere que la distribution de pieces, les penalites et les spectres
7. **Pas de Canvas, SVG, TABLE, jQuery** — uniquement Grid/Flexbox avec Tailwind
8. **Tests a 70%+ de couverture** — commencer a ecrire les tests des le debut du dev
