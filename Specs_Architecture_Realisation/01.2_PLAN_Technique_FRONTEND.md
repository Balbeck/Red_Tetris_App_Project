# Red Tetris — Plan Technique Detaille du Frontend

> Next.js | React | TypeScript | Socket.IO | Tailwind CSS | Redux Toolkit
> Approche Component-First | SPA | Programmation Fonctionnelle

---

## 1. Principes Fondamentaux du Frontend

### 1.1 Regles Imperatives
- **Aucune utilisation de `this`** dans tout le code client (sauf sous-classes Error)
- **Programmation fonctionnelle** : la logique de jeu est implementee exclusivement en fonctions pures
- **SPA** : une seule page HTML, un bundle JS, zero echange de pages HTML avec le serveur
- **Composants fonctionnels React** avec Hooks — jamais de class components
- **Layouts CSS** : uniquement Grid et Flexbox (Tailwind CSS)
- **Interdit** : Canvas, SVG, `<TABLE />`, jQuery, manipulation DOM directe

### 1.2 Philosophie Architecturale — Component-First
L'approche Component-First structure le frontend autour de 3 niveaux de separation :

**Niveau 1 — Pages** : Orchestration pure. Les pages Next.js (App Router) se contentent d'extraire les parametres URL et d'assembler des composants. Zero logique metier.

**Niveau 2 — Layouts** : Chaque composant definit son propre layout. Les layouts de page sont des compositions de layouts de composants. Le positionnement et l'espacement sont geres a ce niveau.

**Niveau 3 — Components** : Toute la logique metier vit ici. Un composant encapsule son comportement, ses interactions socket, son etat local, et son rendu. Il est autonome et reutilisable.

## 3. Pages — Orchestration Pure

### 3.1 Layout Global (`app/layout.tsx`)
**Role** : Wrapper racine de l'application. Ne contient aucune logique metier.

**Responsabilites** :
- Fournir les providers globaux : Redux Provider, Socket Provider
- Importer les styles globaux Tailwind
- Definir les meta tags et le titre de l'application
- Appliquer le fond global de l'application (gradient, animation particules)

### 3.2 Page d'Accueil (`app/page.tsx`)
**Role** : Point d'entree SPA. Orchestre les composants de la page home.

**Responsabilites** :
- Instancier le composant `Home_Component`
- Appliquer le `Layout_Home` pour le positionnement
- Aucune logique — pas de useEffect, pas d'appels socket, pas de state local

### 3.3 Page de Jeu (`app/[room]/[player]/page.tsx`)
**Role** : Page de jeu dynamique. Extraction des params URL et orchestration.

**Responsabilites** :
- Extraire `room` et `player` des parametres de route (Next.js App Router)
- Verifier la validite des parametres (redirection vers `/` si incomplets)
- Orchestrer l'affichage conditionnel :
  - Si `gameStatus === 'waiting'` → afficher `Lobby_Component`
  - Si `gameStatus === 'playing'` → afficher `GameBoard_Component` + `Spectrum_Component` + `NextPiece_Component` + `GameInfo_Component`
  - Si `gameStatus === 'ended'` → afficher `GameOverlay_Component` par-dessus
- Declencher la connexion socket et l'evenement `JOIN_ROOM` au montage (via hook)

## 4. Composants Metier — Specifications Detaillees

### 4.1 Home_Component
**Dossier** : `components/HomeComponents/`

**Logique (`Home_Component.tsx`)** :
- Formulaire avec 2 champs : nom de la room, nom du joueur
- Validation des inputs (non-vide, caracteres autorises, longueur max)
- Navigation programmatique vers `/<room>/<player>` a la soumission
- Gestion d'erreurs via l'evenement `ERROR` du serveur :
  - "Partie en cours" : la room existe mais la game a deja demarre → impossible de rejoindre
  - "Nom deja pris" : un joueur avec le meme pseudo est deja dans la room
  - "Input invalide" : caracteres non autorises dans le nom de room ou de joueur
  - Feedback visuel : notification toast ou message d'erreur sous le formulaire

**Layout (`Layout_Home.tsx`)** :
- Centrage vertical et horizontal (Flexbox)
- Card en glass effect (backdrop-blur, bg-white/10, border, shadow)
- Logo ou titre anime en haut
- Responsive : adapte mobile/desktop

### 4.2 Lobby_Component
**Dossier** : `components/LobbyComponents/`

**Logique (`Lobby_Component.tsx`)** :
- Ecoute les evenements socket : `PLAYER_JOINED`, `PLAYER_LEFT`, `HOST_CHANGED`, `GAME_STARTED`
- Affiche la liste des joueurs connectes dans la room (temps reel)
- Affiche le nom de la room et le nombre de joueurs
- Affiche le bouton "Demarrer" **uniquement si le joueur est le host**
- Emet `START_GAME` quand le host clique sur Demarrer (note : le serveur valide aussi que l'emetteur est bien le host — ne jamais faire confiance au client seul pour l'autorisation)
- Indicateur visuel du host (badge, icone couronne)
- Animation d'entree/sortie quand des joueurs arrivent/partent

**Layout (`Layout_Lobby.tsx`)** :
- Panel central en glass effect
- Liste de joueurs avec avatars/initiales colores
- Bouton proéminent en bas pour le host
- Responsive : empilage vertical sur mobile

### 4.3 GameBoard_Component
**Dossier** : `components/GameBoardComponents/`

**Logique (`GameBoard_Component.tsx`)** :
- Recupere `board` et `activePiece` depuis le store Redux
- Fusionne visuellement la piece active avec le plateau fige (appel a la fonction pure `mergePiece`)
- Genere une grille de 200 cellules (10x20) en iterant sur le tableau 2D
- Chaque cellule recoit un type de bloc (0-8) determine sa couleur
- Affiche optionnellement un "ghost piece" (position d'atterrissage prevue)
- Ne contient **aucune logique de mouvement** — uniquement de l'affichage

**Layout (`Layout_GameBoard.tsx`)** :
- CSS Grid : `grid-template-columns: repeat(10, 1fr)` — 10 colonnes, 20 lignes
- Bordure subtile autour du plateau
- Effet de profondeur (shadow, glass border)
- Taille responsive selon le viewport

### 4.4 Cell_Component
**Dossier** : `components/CellComponents/`

**Logique (`Cell_Component.tsx`)** :
- Recoit un `type` en prop (0 = vide, 1-7 = tetrimino, 8 = penalite)
- Mappe le type vers une classe Tailwind de couleur
- Cellule vide : fond transparent ou tres subtil
- Cellule occupee : couleur vive du tetrimino correspondant
- Cellule penalite : couleur distincte (gris fonce)
- Optionnel : effet de brillance/glass sur les blocs occupes

**Layout (`Layout_Cell.tsx`)** :
- Aspect ratio 1:1 (carre parfait)
- Border subtile entre les cellules
- Border-radius leger pour un look moderne
- Transition de couleur animee

### 4.5 NextPiece_Component
**Dossier** : `components/NextPieceComponents/`

**Logique (`NextPiece_Component.tsx`)** :
- Recupere `nextPiece` depuis le store Redux
- Affiche la piece dans une mini-grille 4x4
- Centre la piece dans la mini-grille selon sa forme
- Met a jour automatiquement quand la next piece change

**Layout (`Layout_NextPiece.tsx`)** :
- Mini-grille CSS Grid 4x4
- Card glass effect avec titre "NEXT"
- Positionnee a droite du plateau principal
- Taille proportionnelle au plateau

### 4.6 Spectrum_Component
**Dossier** : `components/SpectrumComponents/`

**Logique (`Spectrum_Component.tsx`)** :
- Recupere les spectres de tous les adversaires depuis le store Redux (`roomSlice.spectrums`)
- Pour chaque adversaire : affiche une mini-grille de 10 colonnes representant les hauteurs
- Affiche le nom du joueur sous son spectre
- Indicateur visuel du statut : vivant (couleur normale) ou elimine (grise/barre)
- Mise a jour temps reel a chaque reception de `SPECTRUM_UPDATE`

**Layout (`Layout_Spectrum.tsx`)** :
- Conteneur Flexbox horizontal (ou vertical sur mobile)
- Chaque spectre dans une mini-card glass
- Espacement regulier entre les spectres
- Taille adaptative selon le nombre d'adversaires

### 4.7 GameInfo_Component
**Dossier** : `components/GameInfoComponents/`

**Logique (`GameInfo_Component.tsx`)** :
- Affiche le nom de la room
- Affiche le nom du joueur courant
- Affiche le nombre de joueurs restants / total
- Affiche le statut de la partie (en cours, en attente, terminee)
- Optionnel (bonus) : score, lignes effacees, niveau

**Layout (`Layout_GameInfo.tsx`)** :
- Panneau lateral ou barre superieure
- Style glass effect, texte sobre
- Icones pour les differentes informations

### 4.8 GameOverlay_Component
**Dossier** : `components/GameOverlayComponents/`

**Logique (`GameOverlay_Component.tsx`)** :
- S'affiche par-dessus le plateau quand le joueur est elimine ou quand la partie est finie
- Variante "GAME OVER" : quand le joueur est elimine mais d'autres jouent encore
- Variante "VICTORY" : quand le joueur est le dernier survivant
- Variante "DEFEATED" : quand un autre joueur a gagne
- Bouton "Nouvelle Partie" visible **uniquement pour le host**
- Emet `RESTART_GAME` au clic du host
- Animation d'apparition (fade-in, scale-up)

**Layout (`Layout_GameOverlay.tsx`)** :
- Overlay plein ecran avec fond assombri semi-transparent
- Card centrale en glass effect prononce
- Texte large et centre
- Backdrop-blur fort pour l'effet d'immersion

### 4.9 SharedComponents
**Dossier** : `components/SharedComponents/`

**Composants communs reutilisables** :
- `Button_Component.tsx` : Bouton generique avec variantes (primary, secondary, danger), animations hover, taille configurable
- `Input_Component.tsx` : Champ de texte generique avec label, validation, feedback d'erreur
- `Layout_SharedComponents.tsx` : Styles partages (spacings, shadows, glass presets)

## 5. Logique de Jeu — Fonctions Pures (`game/`)

Ce module contient **exclusivement des fonctions pures** : pas d'effets de bord, pas de `this`, resultat deterministe pour les memes entrees. C'est le coeur algorithmique du jeu, execute cote client.

### 5.1 board.ts — Gestion du Plateau
**Fonctions a implementer** :
- `createBoard()` → Cree un plateau vide (tableau 2D de 10x20 rempli de 0)
- `mergePiece(board, piece)` → Retourne un nouveau plateau avec la piece fusionnee (immutable)
- `isValidPosition(board, piece)` → Verifie que la piece ne sort pas des limites et ne chevauche pas un bloc existant
- `addPenaltyLines(board, count)` → Ajoute `count` lignes indestructibles en bas, decale tout vers le haut
- `calculateSpectrum(board)` → Calcule le spectre (tableau de 10 hauteurs) a partir du plateau

### 5.2 pieces.ts — Definition des Tetriminos
**Fonctions et constantes** :
- `TETRIMINOS` : Objet definissant les 7 pieces (I, O, T, S, Z, J, L) avec leurs 4 etats de rotation sous forme de matrices
- `getPieceShape(type, rotation)` → Retourne la matrice de forme d'une piece pour un type et une rotation donnes
- `PIECE_COLORS` : Mapping type → couleur (pour l'affichage Tailwind)
- `getSpawnPosition(type)` → Position initiale de spawn (centree en haut du plateau)

### 5.3 movement.ts — Deplacements et Rotations
**Fonctions a implementer** :
- `moveLeft(board, piece)` → Retourne la piece deplacee a gauche, ou la meme position si invalide
- `moveRight(board, piece)` → Idem a droite
- `moveDown(board, piece)` → Descend la piece d'une ligne
- `rotatePiece(board, piece)` → Rotation de la piece (cycle des 4 etats), avec wall kick basique si collision
- `hardDrop(board, piece)` → Descend la piece instantanement jusqu'a la position la plus basse valide
- `getGhostPosition(board, piece)` → Calcule la position d'atterrissage prevue (pour affichage ghost piece)

### 5.4 lines.ts — Gestion des Lignes
**Fonctions a implementer** :
- `getCompletedLines(board)` → Retourne les indices des lignes ou toutes les cellules sont non-nulles
- `clearLines(board)` → Supprime les lignes completes, fait descendre les lignes superieures, retourne `{ board, linesCleared }`
- `calculatePenalty(linesCleared)` → Calcule le nombre de lignes de penalite a envoyer (`n - 1`)

### 5.5 gravity.ts — Logique de Chute
**Fonctions a implementer** :
- `getGravityInterval(level)` → Retourne l'intervalle en ms entre chaque descente automatique
- `shouldLockPiece(board, piece)` → Determine si la piece doit etre verrouillee (ne peut plus descendre)
- `getNextLevel(linesCleared)` → Calcule le niveau en fonction du total de lignes effacees (pour les bonus)

## 6. Hooks Personnalises (`hooks/`)

### 6.1 useSocket
**Responsabilites** :
- Cree une instance singleton `socket.io-client`
- Se connecte au serveur a l'adresse configuree (env variable)
- Gere la deconnexion automatique au demontage du composant
- Expose une methode `emit(event, data)` typee
- Expose une methode `on(event, callback)` typee
- Gere la reconnexion automatique en cas de perte de connexion

### 6.2 useGameLoop
**Responsabilites** :
- Implemente la boucle de jeu avec `setInterval` (ou `requestAnimationFrame`)
- Gere la vitesse de chute selon le niveau (intervalle dynamique)
- A chaque tick : dispatch l'action `moveDown` dans le store Redux
- Ne demarre que quand `gameStatus === 'playing'`
- Se met en pause quand le joueur est elimine
- Nettoyage propre a la deconnexion/demontage

### 6.3 useKeyboard
**Responsabilites** :
- Ecoute les evenements `keydown` sur `window`
- Mapping des touches vers les actions Redux :
  - `ArrowLeft` → dispatch `moveLeft`
  - `ArrowRight` → dispatch `moveRight`
  - `ArrowUp` → dispatch `rotate`
  - `ArrowDown` → dispatch `softDrop`
  - `Space` → dispatch `hardDrop`
- Empeche le comportement par defaut du navigateur (scroll sur fleches)
- Desactive quand `gameStatus !== 'playing'`
- Gere le debounce/throttle pour eviter les inputs trop rapides

## 7. Store Redux (`store/`)

### 7.1 gameSlice — Etat du Plateau Local
**Etat** :
- `board` : Plateau fige (tableau 2D sans la piece active)
- `activePiece` : Piece en cours de chute (type, position, rotation)
- `nextPiece` : Prochaine piece a jouer
- `status` : 'idle' | 'playing' | 'lost' | 'won'
- `penaltyQueue` : Nombre de lignes de penalite en attente d'application

**Actions** :
- `setPiece(piece)` — Definit la piece active recue du serveur
- `setNextPiece(piece)` — Stocke la prochaine piece
- `moveLeft / moveRight / rotate / softDrop / hardDrop` — Actions de mouvement (appellent les fonctions pures de `game/`)
- `lockPiece` — Fige la piece active sur le plateau
- `clearLines` — Detecte et supprime les lignes completes
- `addPenaltyLines(count)` — Applique les penalites recues
- `setGameStatus(status)` — Change le statut du jeu
- `resetGame` — Remet le plateau a zero pour une nouvelle partie

### 7.2 roomSlice — Etat de la Room
**Etat** :
- `roomName` : Nom de la room actuelle
- `playerName` : Nom du joueur courant
- `players` : Liste des joueurs dans la room (nom, statut alive/dead)
- `isHost` : Booleen — ce joueur est-il le host ?
- `spectrums` : `Record<string, number[]>` — Spectres de tous les adversaires
- `gameStatus` : 'waiting' | 'playing' | 'ended'
- `winner` : Nom du vainqueur (null si partie en cours)

**Actions** :
- `setRoomState(state)` — Initialisation complete a la connexion
- `playerJoined(player)` — Un joueur a rejoint
- `playerLeft(player)` — Un joueur a quitte
- `updateSpectrum({ player, spectrum })` — MAJ du spectre d'un adversaire
- `playerEliminated(playerName)` — Un joueur elimine
- `gameStarted` — Transition vers l'etat playing
- `gameEnded(winner)` — Fin de partie
- `hostChanged(newHost)` — Transfert du role de host

### 7.3 uiSlice — Etat UI
**Etat** :
- `currentScreen` : 'home' | 'lobby' | 'game' | 'gameOver'
- `notifications` : Messages temporaires a afficher
- `isConnected` : Statut de la connexion socket

### 7.4 socketMiddleware — Pont Redux ↔ Socket.IO
**Principe** : Un middleware Redux qui intercepte certaines actions et les traduit en evenements socket, et inversement.

**Actions Redux → Emissions Socket** :
- `lockPiece` → apres traitement, emet `REQUEST_PIECE`, `UPDATE_SPECTRUM`, et optionnellement `LINES_CLEARED`
- `setGameStatus('lost')` → emet `GAME_OVER_PLAYER`

**Evenements Socket → Actions Redux** :
- `ROOM_STATE` → dispatch `setRoomState`
- `PLAYER_JOINED` → dispatch `playerJoined`
- `PLAYER_LEFT` → dispatch `playerLeft`
- `HOST_CHANGED` → dispatch `hostChanged`
- `GAME_STARTED` → dispatch `gameStarted` + `setGameStatus('playing')`
- `NEW_PIECE` → dispatch `setPiece` ou `setNextPiece`
- `SPECTRUM_UPDATE` → dispatch `updateSpectrum`
- `PENALTY_LINES` → dispatch `addPenaltyLines`
- `PLAYER_ELIMINATED` → dispatch `playerEliminated`
- `GAME_OVER` → dispatch `gameEnded`
- `ERROR` → dispatch notification UI (ex: "Room en cours", "Nom deja pris")

## 8. Socket Client (`socket/`)

### 8.1 socket.ts — Instance Singleton
- Cree et exporte une instance unique de `socket.io-client`
- Configuration : URL du serveur (depuis variable d'environnement), options de reconnexion
- Ne se connecte pas automatiquement — la connexion est declenchee par le hook `useSocket`

### 8.2 events.ts — Constantes d'Evenements
- Re-exporte les constantes definies dans `shared/events.ts`
- Garantit une source de verite unique entre client et serveur
- Typage fort des evenements et de leurs payloads

## 10. Routing — SPA avec Next.js App Router

### 10.1 Routes Definies
| Route | Page | Description |
|-------|------|-------------|
| `/` | `app/page.tsx` | Accueil — formulaire room/player |
| `/<room>/<player>` | `app/[room]/[player]/page.tsx` | Jeu — lobby puis plateau |

### 10.2 Navigation
- Navigation **programmatique** via `useRouter().push()` de Next.js
- Pas de rechargement de page (SPA)
- Validation des parametres URL au montage de la page de jeu
- Redirection vers `/` si parametres invalides
- L'URL sert de **source de verite** pour la room et le nom du joueur

### 10.3 Gestion de l'Historique
- Utilisation du router Next.js (basé sur le History API)
- Retour arriere (bouton navigateur) → retour a la page d'accueil
- Le sujet recommande l'utilisation de routeurs type BrowserRouter/MemoryRouter — avec Next.js App Router, c'est natif

## 11. Flux d'Execution Frontend — Cycle de Vie

### 11.1 Arrivee sur la Page d'Accueil
1. L'utilisateur accede a `http://server:port/`
2. La page affiche le formulaire `Home_Component`
3. L'utilisateur saisit un nom de room et un pseudonyme
4. Au clic sur "Rejoindre", navigation vers `/<room>/<player>`

### 11.2 Arrivee sur la Page de Jeu
1. Extraction des params `room` et `player` depuis l'URL
2. Initialisation de la connexion socket via `useSocket`
3. Emission de `JOIN_ROOM` avec `{ room, playerName }`
4. Reception de `ROOM_STATE` → mise a jour du store Redux
5. Affichage du `Lobby_Component` (statut = 'waiting')

### 11.3 Demarrage de la Partie
1. Le host clique sur "Demarrer" → emission de `START_GAME`
2. Reception de `GAME_STARTED` → transition vers l'ecran de jeu
3. Emission de 2x `REQUEST_PIECE` :
   - 1ere `NEW_PIECE` recue → stockee directement comme `activePiece` (piece immediatement jouable)
   - 2eme `NEW_PIECE` recue → stockee comme `nextPiece` (affichee dans le panneau Next)
4. Activation du `useGameLoop` et `useKeyboard`
5. La piece active commence a descendre automatiquement
6. Par la suite, a chaque `lockPiece` : `nextPiece` devient `activePiece`, et un nouveau `REQUEST_PIECE` est emis pour remplir `nextPiece`

### 11.4 Boucle de Jeu
1. **Chaque tick** (intervalle de gravite) : la piece descend d'une ligne (`moveDown`)
2. **Input clavier** : deplacement/rotation/drop instantane
3. **Pose de piece** : quand la piece ne peut plus descendre → `lockPiece` :
   a. Fusion de la piece avec le plateau
   b. Detection et suppression des lignes completes
   c. Si lignes effacees : emission `LINES_CLEARED` + calcul penalites
   d. Calcul et emission `UPDATE_SPECTRUM`
   e. Emission `REQUEST_PIECE` pour la piece suivante
   f. Reception de `NEW_PIECE` → la piece suivante devient active
4. **Reception de penalites** : `PENALTY_LINES` → ajout de lignes indestructibles en bas
5. **Elimination** : si la nouvelle piece ne peut pas spawn → `GAME_OVER_PLAYER`

### 11.5 Fin de Partie
1. Reception de `GAME_OVER` avec le nom du vainqueur
2. Affichage du `GameOverlay_Component`
3. Si le joueur est host : affichage du bouton "Nouvelle Partie"
4. Au clic host : emission `RESTART_GAME` → retour au lobby

## 12. Strategie de Tests Frontend

### 12.1 Tests Fonctions Pures (`game/`) — Priorite Maximale
Ces tests representent le meilleur ratio couverture/effort car les fonctions pures sont deterministes et sans effets de bord.

**board.ts** :
- `createBoard()` : verifier dimensions 10x20, toutes les cellules a 0
- `mergePiece()` : verifier que la piece est correctement fusionnee
- `isValidPosition()` : tester limites gauche/droite/bas, chevauchement avec blocs existants
- `addPenaltyLines()` : verifier decalage vers le haut, lignes indestructibles en bas
- `calculateSpectrum()` : verifier le calcul des hauteurs par colonne

**movement.ts** :
- `moveLeft/Right()` : deplacement normal + blocage aux murs
- `rotatePiece()` : tester les 4 rotations pour chaque type de piece
- `hardDrop()` : verifier l'atterrissage correct
- `getGhostPosition()` : verifier la position prevue

**lines.ts** :
- `getCompletedLines()` : plateau vide, une ligne pleine, plusieurs lignes
- `clearLines()` : verifier la suppression et la descente des lignes superieures
- `calculatePenalty()` : verifier le calcul n-1

**pieces.ts** :
- Verification de la coherence des matrices de rotation pour les 7 pieces

### 12.2 Tests Composants React — Priorite Moyenne
- `GameBoard_Component` : rend bien 200 cellules (10x20)
- `Lobby_Component` : affiche le bouton Start uniquement si `isHost === true`
- `Spectrum_Component` : affiche correctement les spectres des adversaires
- `GameOverlay_Component` : affiche le bon message selon le statut (won/lost)
- `Cell_Component` : affiche la bonne couleur selon le type

### 12.3 Tests Hooks — Priorite Moyenne
- `useKeyboard` : verifie que les bonnes actions sont dispatches
- `useGameLoop` : verifie le demarrage/arret selon le gameStatus

### 12.4 Tests Redux — Priorite Moyenne
- `gameSlice` : verifier les reducers (moveLeft change bien la position, clearLines supprime les bonnes lignes)
- `roomSlice` : verifier les reducers (playerJoined ajoute bien, hostChanged met a jour)
- `socketMiddleware` : verifier les mappings actions ↔ events
